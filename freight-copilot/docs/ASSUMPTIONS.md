# Freight Copilot — Assumptions Register

This document records every significant unknown encountered during design and the provisional assumption chosen. Revisit these when circumstances change or confirmation becomes available.

---

## Assumptions Table

| # | Unknown | Provisional Assumption | Why | How to Confirm |
|---|---|---|---|---|
| A-01 | Which database engine to use as the default | **SQLite** (single file, `freight.db`) | One operator, low concurrency, zero setup friction, local-first deployment | Reassess if concurrent users are needed or if deployment moves to a server; switch `DATABASE_URL` to PostgreSQL |
| A-02 | When to switch from SQLite to PostgreSQL | Switch when multi-user access or a hosted/cloud deployment is required | SQLite handles tens of concurrent readers without issue for a single operator; write contention only matters with multiple simultaneous users | Monitor for write contention errors; plan Postgres migration before launching multi-user |
| A-03 | Which UI technology for the MVP | **Server-rendered Jinja2 + Tailwind (CDN)** | Fastest path to a working dashboard without a separate build pipeline; no npm/Node required to run the backend | Confirm with operator that the MVP UI meets daily-use needs before investing in Next.js |
| A-04 | When to build the Next.js frontend | **Roadmap — after MVP validates core workflows** | Next.js adds a second deployment artefact and build pipeline; defer until product-market fit is clearer | Decide when operator requests richer interactivity or mobile support |
| A-05 | Whether live market rates are available | **No live rates** — pricing is rule-based using configurable EUR/km bands | No integration with a live rate feed is planned for MVP; rule-based pricing is sufficient for decision support | Integrate a market rate API (e.g., road freight index feeds) when available and budgeted |
| A-06 | Baseline EUR/km rate for pricing | **~1.0–1.5 EUR/km for standard FTL (full truckload) on main European corridors** | Representative mid-market range based on industry knowledge as of 2025; varies by corridor, season, load type | Calibrate against the operator's own historical invoices during onboarding; update the pricing config table |
| A-07 | Baseline EUR/km rate for LTL | **~2.0–3.5 EUR/km (LTL premium over FTL)** | LTL consolidation and handling adds cost; rough multiplier of 2× FTL is a common rule of thumb | Same as A-06 — operator's own data overrides any default |
| A-08 | Whether TIMOCOM has an official API | **Assumed yes** — adapter interface is designed for an official API | TIMOCOM is a major exchange that has offered API access; the adapter is a placeholder pending API credential acquisition | Obtain TIMOCOM API documentation and credentials; implement the real adapter |
| A-09 | Whether Trans.eu has an official API | **Assumed yes** — adapter interface is designed for an official API | Trans.eu is a major exchange that has offered API access | Obtain Trans.eu API documentation and credentials; implement the real adapter |
| A-10 | What data TIMOCOM/Trans.eu APIs actually expose | **Unknown in detail** — adapters return mock data matching the internal schema | No API credentials or documentation reviewed during MVP build | Review official API documentation when credentials are obtained; adjust adapter schemas accordingly |
| A-11 | Whether demo carrier data is real | **All demo carriers are synthetic** | Using real company names or registration numbers without consent creates legal and privacy risk | Seed data is clearly labelled as synthetic; replace with real carrier directory during operator onboarding |
| A-12 | CMR and document field requirements | Standard CMR fields as defined by the CMR Convention (Geneva, 1956) and common carrier practice | The convention defines the mandatory fields; practical additions (e.g., VAT numbers) vary by country | Review with the operator against their actual document templates |
| A-13 | Email protocol | **IMAP** for reading, **SMTP** for sending | Most business email is accessible via IMAP/SMTP; Exchange/O365 IMAP is also standard | Confirm the operator's email provider and credentials; adjust `EmailAdapter` if OAuth2 (Microsoft Graph) is required |
| A-14 | Number of concurrent users at launch | **One** (the operator themselves) | The business description specifies a single forwarder | Re-evaluate if the operator has assistants or the tool is offered to multiple forwarders |
| A-15 | Whether an LLM is required for the product to be useful | **No** — the product is fully operational with `DeterministicProvider` | Rule-based agents cover all critical workflows; LLM improves draft quality and classification but is optional | Gather operator feedback on draft quality; enable LLM if deterministic drafts are insufficient |
| A-16 | Which Anthropic models to use when LLM is enabled | **Primary:** `claude-opus-4-8`; **Subagents:** `claude-sonnet-4-6` | Opus for complex reasoning (pricing, risk); Sonnet for faster, cheaper subagent tasks | Adjust model selection based on latency, cost, and quality benchmarks once real usage data is available |
| A-17 | Whether GPS/ETA tracking data is available | **Not available for MVP** — Transport Monitoring relies on manual status updates | No tracking API integration is planned for MVP | Integrate a telematics or tracking provider (e.g., carrier-provided GPS links, FourKites, project44) when funded |
| A-18 | Compliance rules to enforce | **Cabotage (EU Regulation 1072/2009), ADR (hazardous goods), EC 561/2006 (driving hours)** | These are the three most commonly encountered compliance constraints in European road freight | Review with the operator for any company-specific internal rules to add; consult legal counsel for jurisdiction-specific nuance |
| A-19 | Currency conversion for PLN/EUR calculations | **No live FX rates** — a configurable static rate is used (default: operator-supplied) | Live FX integration is disproportionate for MVP; daily rate manually updated is sufficient | Integrate a live FX feed (e.g., NBP API for PLN) when exchange-rate sensitivity becomes material |
| A-20 | Data retention and GDPR scope | **All data stays local** — no cloud sync, no third-party analytics | Local-first deployment minimises data controller obligations; no personal data leaves the operator's machine | Review with the operator if cloud deployment or multi-tenancy is introduced; conduct a DPIA at that point |
| A-21 | Test coverage target for MVP | **Unit tests for all agent logic; integration tests for key API routes** | Agent correctness is the highest-risk area; integration tests catch wiring errors | Measure coverage with `pytest --cov`; target ≥80% line coverage on `agents/` module |
| A-22 | Deployment environment | **Local machine (Linux/macOS), Python 3.11+** | Operator runs from their own workstation; no cloud infra required | Confirm OS and Python version with operator; test on Windows if needed |
